Уменьшение связанности

  • Классы А и Б – связаны тогда, когда нельзя понять как А устроен, не зная устройства Б.

Связанности мало, значит меньше нужно помнить, значит легче понять код.
Но связанность уменьшается ‘переходниками’.
Больше ‘переходников’, значит больше кода, значит труднее понять код.

Жажда скорости

‘Переходники’ уменьшают производительность игры, но ускоряют прототипирование.
Легче сделать интересную игру быстрой (убрать ‘переходники’), чем быструю игру интересной.
Говнокод пишется ещё быстрее чем код с ‘переходниками’,
поэтому, его можно использовать для прототипирования.

  • Используем шаблоны, пока не получится что-то интересное, а затем оптимизируем.
  • Прототипный говнокод, после проверки идей, необходимо выбрасывать.


Хорошо оптимизированный код часто

  • прост для восприятия
  • покрывает малую область использования
  • долго придумывается (от структур, через алгоритм)
  • краток


Элегантный код – неплохо оптимизирован и устойчив (покрывает большую облать использования).

Недооценённый Command

Команда - это ‘овеществление’ вызова метода
Bob Nystrom

bool closed = false;

inline void close()
{
    closed = true;
}

struct COMMAND {
    virtual void Exec(ACTOR& a) = 0;
    virtual ~COMMAND() = default;
};
using pCOMMAND = std::unique_ptr<COMMAND>;

// Новые команды добавляются совсем просто
struct NULL_OBJ_COMMAND: public COMMAND {void Exec(ACTOR& a) override {}};
struct JUMP_COMMAND:     public COMMAND {void Exec(ACTOR& a) override {a.Jump();}};
struct SIT_DOWN_COMMAND: public COMMAND {void Exec(ACTOR& a) override {a.Sit();}};
struct CLOSE_COMMAND:    public COMMAND {void Exec(ACTOR& a) override {close();}};

struct INPUT_HANDLER {

    pCOMMAND button_W;
    pCOMMAND button_S;
    pCOMMAND button_N;
    pCOMMAND button_Q;

    COMMAND& HandleInput (KEY pressed)
    {
        switch (pressed) {
            case KEY::W: return button_W.get();
            case KEY::S: return button_S.get();
            case KEY::N: return button_N.get();
            case KEY::Q: return button_Q.get();
            default return pass.get ();
        }
    }
private:
    static pCOMMAND pass;
};
pCOMMAND INPUT_HANDLER::pass = new NULL_OBJ_COMMAND();

auto main () -> int
{
    // Ставим настройки
    auto input = INPUT_HANDLER {
        new JumpCommand(),    // для Key::W
        new SitDownCommand(), // для Key::S
        new NullObjCommand(), // для Key::N
        new CloseCommand(),   // для Key::Q
    };

    Actor actor;
    while (!closed) {
        puts("Press W to jump, S to sit-down and Q to exit ^_^");
        inpt.HandleInput(Keyboard::getKey ())->exec (actor);
    }
}
  • Можно запоминать команды в очередь (событий), разрывая связь между заказчиком и исполнением.
  • Можно добавить метод undo в Command и хранить историю комад в списке.
  • Если методы класса Actor вызывают методы своих полей, то это шаблон ‘Цепочка ответственности’.






Два особо удачных для игр

- Flyweight

Есть объект внутреннего класса, данные которого повторяются у кучи объектов внешнего (контекстно-зависимого) класса.

// Внутренний класс с тяжёлой текстурой
// и данными, которые не нужно менять
struct TERRAIN {
    TERRAIN (const TEXTURE& texture, int weight)
     : texture_ (texture)
     , weight_ (weight)
    {}

    int GetWeight () const
    {
        return weight_;
    }

    TEXTURE* GetTexture () const
    {
        return &texture_;
    }
private:
    int weight_;
    TEXTURE texture_;
};
// Внешний класс
struct WORLD {

    WORLD ()
    {
        for (size_t x=0, x < width, ++x) {     
            for (size_t y=0, y < width, ++y) {
                tiles_[x][y] = (random (10) == 0)? &hillTerrain_ : &grassTerrain_;
            }
        }
        size_t x = random (width);

        for (size_t y=0, y < width, ++y) {
            tiles[x][y] = &riverTerrain_;
        }
    }

    TERRAIN* GetTile(size_t x, size_t y)
    {
        return tiles_[x][y];
    }
private:
    // По одному тяжёлому объекту, на каждый тип покрытия
    TERRAIN grassTerrain_{GRASS_TEXTURE, 1};
    TERRAIN hillTerrain_ {HILL_TEXTURE,  3};
    TERRAIN riverTerrain_{RIVER_TEXTURE, 2};

    const size_t width = 10000, height = 20000;

    Terrain* tiles_[width][height];
};

  1. Экономим память
  2. Экономим время на отладку

- Observer

enum EVENT {
    TheRoofIsOnFire,
    SomeAnotherEvent,
};
// Интерфейсы наблюдаемого и наблюдателя
class OBSERVER {
public:
    virtual ~OBSERVER() = default;
    virtual void OnNotify(EVENT event) = 0;
};

class SUBJECT {
protected:
    void Notify(EVENT event)
    {
        for (auto& it : observers) {
            it->OnNotify (event);
        }
    }

    std::vector<OBSERVER*> observers;
};
// --------------------------------------------
class ACHIEVEMENT : public OBSERVER { 
public:
    virtual void OnNotify(EVENT event)
    {
        switch(event) {
            case EVENT::TheRoofIsOnFire:

                if (!fireAchivementUnlocked) {
                    puts("[We don't need no water, let the mfkr burn!] Achievement unlocked");
                    fireAchivementUnlocked = true;
                }

            break;
            default: break;
        }
    }
private:
    bool fireAchivementUnlocked = false;
};

class PHYSICS : public SUBJECT {
public:
    void UpdateFire()
    {
        if (random(10) > 8) {
            Notify(Event::TheRoofIsOnFire);
        }
    }

    void Subscribe(OBSERVER* o)
    {
        observers.push_back(o);
    }
};

auto main() -> int
{
    ACHIEVEMENT achivement;
    PHYSICS roof;

    roof.Subscribe (&achivement);
    // 1. можно подписать ачивку на кучу других вещей
    // 2. можно подписать на горящую крышу, воспроизведение звука, анимации и т.д.

    for (size_t idx = 0; idx < 1000; ++idx) {
        roof.UpdateFire ();
    }
}

  1. Легко создаются связи между объектами.
  2. До попы сложная отладка. Трудно понять, что с чем связано.

- State

В gamedev широко применяется подвид автоматов – final state machine (бывают ещё hierarchical state machine, pushdown automata и т.д.). FSM позволяет не писать код с огромным количеством if-овых ветвлений и заставляет чётко выделить состояния и условия изменения.

Это легче воспринимать и легче спроектировать, но FSM – автомат не Тьюринг полный и подходит для разработки простого AI (для чуть более сложного, используют behavior tree и planning systems).

Кроме того, кол-во состояний увеличивается во много раз, даже от небольших изменений.

Можно реализовать FSM используя switch, или if с elif и else:

enum class STATE {
    Standing,
    Ducking,
    Jumping,
    Diving,
};

class PLAYER {
public:
    void HandleInput(INPUT input)
    {
        switch (state) {
            case STATE::Standing:
            if (input == INPUT::Press::B) { // условие перехода
                state = STATE::Jumping;
                this.SetAnimation(ANIMATION::Jump);
            }
            break;
            /* то же самое для всех остальных состояний */
            default: break;
        }
    }
};

но так, мы быстро наткнёмся на смешивание состояний. Так как они лежат в одном месте, они начнут ссылаться на одни и те же поля. Это будет трудно отлаживать и расширять, поэтому и существует шаблон State.

Суть в том, чтобы заменить switch на виртуальную таблицу, которая генерируется автоматически.

// Интерфейс состояний
class PLAYER_STATE {
public:
    virtual ~PLAYER_STATE() = default;
    virtual void HandleInput(PLAYER& player, INPUT input) {}
    virtual void Update(PLAYER& player) {}
};
// Теперь можно создавать состояния по мере надобности
class DUCKING_STATE : public PLAYER_STATE {
public:
    DUCKING_STATE() = default;

    void HandleInput(PLAYER& player, INPUT input) override
    {
        if (input == RELEASE_DOWN) {
            player.SetState (PLAYER::standing);
            player.SetAnimation(ANIMATION::Stand);
        }
    }

    void Update(PLAYER& player) override
    {
        ++chargeTime;
        if (chargeTime > MAX_CHARGE) {
            player.SuperBomb();
        }
    }
private:
    // Имеет отношение только к этому состоянию
    int chargeTime{0};
};
// ... Тут классы других состояний ...


// Теперь, тупо дилегируем обработку в 'состояние'
class PLAYER {
public:
    static STANDING_STATE standing;
    // ... Остальные объекты состояния ...

    virtual void HandleInput(INPUT input)
    {
        state->HandleInput(*this, input);
    }

    virtual void Update()
    {
        state->Update(*this);
    }

    virtual void SetState(PLAYER_STATE& state)
    {
        this.state = &state;
    }
    // ... Всё остальное, что должен уметь Player ...
private:
    PLAYER_STATE* state;
};

Интересные факты:

  1. Если состояния имеют по одному виртуальному методу и не имеют никаких полей, то можно обойтись ссылками на функции состояния.
  2. Можно впихивать статические экземпляры состояний в какой-нибудь класс, если нам не важно, что этими состояниями будут пользоваться много других FSM.
  3. Можно динамически создавать состояния под кажное изменение. В этом случае, у каждой FSM гарантированно будет по своему личному состоянию, но это будет приводить к фрагментации памяти и ударит по скорости.

Если PLAYER, из нашего примера, будет иметь ссылки на два состояния двух FSM, то такая система называется Concurent State Machines.

CSM предпологают, возможность ‘поглащать’ input некоторыми состояниями. Помимо этого, для взаимодействия конкурентных состояний, используют грубые if проверки, что до жути всё усложняет, приводит к багам, но как-то работает.

Если у классов-наследников PLAYER_STATE (т.е. суперсостояний) будут виртуальные функции и наследники (т.е. уже подсостояния), то такой автомат называется: Hierarchical State Machine. При таком подходе, самые рабочие состояния будут находиться в листьях цепочки наследований. Иногда, они будут дилегировать работу своим суперсостояниям.

Если иерархия суперсостояний и подсостояний хорошо известна (и, скорее всего, не будет меняться), то для соблюдения иерархии можно использовать стек состояний, в котором подсостояния – вверху, а суперсостояния – внизу.

HSM нужен для сокращения повторяющегося кода (но классов меньше не становится), за это, сложеность выполнения программы увеличивается.

Если там, где в FSM у нас находится ссылка, будет находиться стек ссылко, то такое называется Pushdown Automata

Pushdown Automata нужен, когда автомату часто необходимо перемещаться назад, по истории предыдущих состояний.






Переоценённый Singleton

class MAIN_MODE {
public:
    static MAIN_MODE& instance()
    {
        if (pInstance == nullptr) pInstance = new MAIN_MODE(); // ленивая инициализация

        return *pInstance;
    }

private:
    MAIN_MODE() = default;

    static MAIN_MODE* pInstance;
};

MAIN_MODE* MAIN_MODE::pInstance{nullptr};
  • По сути, это глобальная переменная, так что обчно х.з. в каком она находится состоянии
    • делает код менее понятной
    • усиливыает связанность (чтобы что-то поменять, приходится думать обо всех этих глобальных штуках)
    • приводит к гонкам за данными

  • Решает две проблемы (копия одна и она легко доступна), даже если вам надо решить одну

    • чтобы экземпляр был один, можно сделать флаг instantiated в private, а в конструкторе звать assert(!instantiated)
    • чтобы доставать было проще, можно:
    1. каждый раз, всюду делать инъекцию необходимых инструментов в функцию
    2. сделать сквозное связывание, т.е. бахнуть всюду нужный функционал в protected самого базового класса (бьёт по производительности)
    3. воспользоваться глобальным объектом, уже написанным до вас (бывают World, Game, AppMode, вот это вот всё)
    4. использовать шаблон Service Locator

  • Будет плохо, если ленивая инициализация чего-то большого пойдёт во время разгара online битвы.

Так что, singleton вам не нужен.
